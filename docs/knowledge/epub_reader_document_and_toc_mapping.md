# EPUB 正文文档与目录项映射

## 文档定位

这是一篇面向开发者的背景知识文档，用来解释为什么阅读器实现里通常需要把“正文文档”和“目录项”建成两层模型，而不是把它们混成一个 `Chapter`。

它的目标是：

- 帮助理解正文文件、阅读顺序和目录项之间的边界
- 帮助理解为什么“一个文件不一定等于一个章节”
- 帮助理解阅读器内部模型通常为什么要拆层

## 为什么不能把“章节”当成唯一模型

很多初学实现会自然地想要一个模型：

```text
Chapter
- title
- content
- index
```

但 EPUB 里“章节”这个词往往混合了好几种不同概念：

- 用户看到的目录节点
- 正文阅读顺序单元
- 实际资源文件
- 跳转目标
- 阅读进度锚点

如果把这些职责塞进一个模型，后面很容易出现边界混乱。

## 实现里更稳的拆分方式

很多阅读器会至少区分下面几层。

### 1. Resource

资源文件层。

表示这本 EPUB 里有哪些实际资源：

- XHTML
- CSS
- 图片
- 字体

### 2. SpineItem

阅读顺序层。

它表达的是：

- 哪些资源参与正文顺序
- 它们按什么顺序读

它更接近“阅读顺序单元”，而不一定等于用户理解的章节。

### 3. ReaderDocument

正文渲染层。

很多实现会把真正用于阅读器展示和跳转的基本单元单独抽出来。这个单元常常更接近：

- 一个可渲染正文文档
- 一个稳定的文件级跳转目标
- 一个可保存进度的位置载体

### 4. TocItem

用户目录层。

它表达的是：

- 目录标题
- 层级关系
- 指向的位置

它更接近用户眼中的“章节”。

## 常见的映射关系

### 情况 1. 一个 TOC 项对应一个正文文件

这是最直观也最容易处理的情况。

```text
TocItem A -> ReaderDocument 3
```

### 情况 2. 多个 TOC 项对应同一个正文文件

例如一个文件里有：

- 第三章
- 第三章第一节
- 第三章第二节

这时可能出现：

```text
TocItem A -> chapter3.xhtml#top
TocItem B -> chapter3.xhtml#part1
TocItem C -> chapter3.xhtml#part2
```

底层文件只有一个，但用户看到是多个目录项。

### 情况 3. 一个逻辑章节跨多个正文文件

有些很长的章节会被制作成：

```text
chapter5-1.xhtml
chapter5-2.xhtml
chapter5-3.xhtml
```

这时用户会认为这是“第五章”，但底层可能已经是多个阅读单元。

## 为什么这会直接影响导航设计

如果你把目录项直接当成正文单元，就会立刻遇到问题：

- 上一章/下一章到底按哪个顺序移动
- 同一文件内多个目录项是否都能单独点击
- 阅读进度应该记在目录项上还是文档上
- UI 目录标题和正文标题冲突时谁优先

因此更稳的思路通常是：

- 正文展示和线性导航建立在正文文档层
- 目录展示建立在 TOC 层
- 两者通过映射关系连接

## 一个实用的心智模型

可以把它想成三层：

```text
spine -> ReaderDocument -> TocItem
```

更准确一点说：

- `spine` 帮你决定正文顺序
- `ReaderDocument` 帮你决定稳定渲染和基础跳转单元
- `TocItem` 帮你决定用户看到什么目录入口

用户口中的“跳到第三章”，很多时候真正做的是：

1. 找到对应 `TocItem`
2. 解析出它指向的文件和锚点
3. 落到某个 `ReaderDocument`
4. 再决定是否需要文内二次定位

## 哪些功能主要依赖哪一层

- 上一章/下一章：通常更依赖正文文档顺序
- 目录树展示：更依赖 `TocItem`
- 书签与阅读进度：通常依赖更细粒度的定位模型，但至少要能落回正文文档
- 精确跳转：通常要在“正文文档 + 锚点”层完成

## 常见误区

### 1. 用一个数组同时当正文和目录

短期实现很快，后期非常难稳。

### 2. 用 TOC 顺序直接驱动正文渲染顺序

有些书能工作，但一旦 TOC 不完整就会出问题。

### 3. 假设一个文件一定只对应一个标题

现实 EPUB 里并不成立。

### 4. 把目录标题直接当正文标题

用户入口标题和正文自身标题有时是一致的，但并不保证总是一致。

## 如果你正在设计阅读器内部模型

一个比较稳的最小拆分通常包括：

- `ReaderDocument`
- `TocItem`
- `Locator`

必要时再加：

- `Resource`
- `SpineItem`

这比“一切都塞进 Chapter”更容易演进。
