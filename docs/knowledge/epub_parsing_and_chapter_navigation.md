# EPUB 解析与章节跳转实现

## 文档定位

这是一篇面向开发者的背景知识文档，目标是帮助人快速建立对 EPUB 解析、目录、阅读顺序和章节跳转的整体认识。

它的作用是：

- 帮助开发者理解 EPUB 相关概念
- 帮助开发者判断章节跳转能力为什么难做稳
- 作为学习和沟通时的高层说明

它不承担的作用是：

- 不作为本仓库 agent 默认必须阅读的执行规范
- 不直接替代项目中的实施约束文档
- 不试图覆盖某个具体渲染框架的全部落地细节

如果某个任务需要严格的项目实施口径，应优先查看 `docs/problem/` 下的约束文档。

## 核心认识

EPUB 不是一本“完整排版好的书”，而更像一个压缩包，里面包含：

- HTML/XHTML 正文
- CSS 样式
- 图片、字体等资源
- 元数据
- 阅读顺序定义
- 目录定义

多数阅读软件的“章节跳转很完善”，本质上并不是在运行时去猜哪一段是章节，而是优先使用 EPUB 自身已经提供的结构信息：目录、文档顺序、锚点、内部链接。

## EPUB 的常见结构

`.epub` 文件本质上是 ZIP。常见关键内容包括：

- `mimetype`
- `META-INF/container.xml`
- 一个 `*.opf` 包文件
- 多个正文 `xhtml/html`
- 图片、字体、CSS
- 目录文件：EPUB 2 常见 `toc.ncx`，EPUB 3 常见带 `nav` 属性的导航文档，文件名经常是 `nav.xhtml`

其中最重要的是 `opf`，通常定义三类信息：

- `manifest`：这本书包含哪些资源文件
- `spine`：正文的阅读顺序
- metadata：书名、作者、语言等元数据

可以简单理解为：

- `manifest` = 文件清单
- `spine` = 阅读顺序
- `nav/toc` = 给用户看的目录

## 阅读软件通常如何解析 EPUB

常见解析流程大致如下：

1. 解压 `.epub`
2. 读取 `META-INF/container.xml`
3. 找到 `content.opf` 或等价的包文件
4. 解析 `manifest` 和 `spine`
5. 建立资源索引：`id -> href`
6. 解析目录文件：EPUB 3 导航文档或 `toc.ncx`
7. 按 `spine` 顺序加载章节 HTML
8. 统一处理相对路径、图片、CSS、锚点和内部链接

所以“章节跳转”做得是否稳定，关键并不在 UI，而在于底层有没有建立好以下索引：

- 章节顺序索引
- 目录项到文件/锚点的映射
- 当前阅读位置到章节的反向映射

## 章节跳转为什么能做得很完善

成熟阅读器一般不会只依赖一份信息，而是同时维护三套定位能力。

### 1. 目录定位

来源通常是 EPUB 3 导航文档或 `toc.ncx`。

每个目录项一般会对应一个 `href`，例如：

```text
Text/chapter3.xhtml#section2
```

这意味着目录项不仅能定位到某个文件，还能定位到该文件内部的某个锚点。

### 2. 阅读顺序定位

来源是 `spine`。

即使目录缺失、不完整，阅读器依然知道当前内容的前后顺序，可以实现：

- 上一章
- 下一章
- 按顺序预加载内容

### 3. 文内锚点定位

正文 XHTML 中通常会包含类似：

```html
<h1 id="chapter5">第五章</h1>
```

当目录项指向 `chapter5.xhtml#chapter5` 时，阅读器会：

1. 打开 `chapter5.xhtml`
2. 渲染成 DOM 或中间结构
3. 在渲染完成后滚动到 `#chapter5`

因此，一个目录点击动作通常不是“打开第 N 章”这么简单，而是：

- 找到目录项对应的 `href`
- 解析出目标文档和目标 fragment
- 打开目标文档
- 在文档内定位到 fragment 对应的锚点

## 章节跳转通常依赖哪些 EPUB 信息

如果只讨论“目录从哪里来”，实现时通常按以下顺序处理：

1. EPUB 3 导航文档，通常优先
2. `toc.ncx`，EPUB 2 常见
3. 如果目录缺失或质量较差：
   - 退化使用 `spine`
   - 或从正文标题中推断章节边界

但如果讨论“正文阅读顺序从哪里来”，通常还是优先看 `spine`，再考虑 fallback。

这也是为什么不同书籍在同一个阅读器里的目录体验差异很大。很多问题不在阅读器，而在于书本身制作质量不高：

- 目录缺失
- 目录链接错误
- 锚点不存在
- 文档顺序和目录不一致

## “章节”不一定等于“一个文件”

这是理解 EPUB 很重要的一点。

在实际书籍中，可能出现：

- 一个章节对应一个 XHTML 文件
- 多个章节放在同一个 XHTML 文件中
- 一个很长的章节被拆成多个 XHTML 文件

因此成熟阅读器通常不会把“章节”简单等同于“文件”，而是区分三类概念：

- `spine item`：阅读顺序单元
- `toc item`：用户看到的目录项
- `anchor`：文内定位点

用户眼中的“章节”很多时候更接近 `toc item`，而不是底层资源文件。

## 为什么阅读器能准确恢复到上次阅读位置

成熟阅读器通常不会只保存“第几章”，而会保存更细粒度的位置。常见方式包括：

- `spineIndex + offset`
- `href + anchor`
- `href + 文本偏移`
- `EPUB CFI`

如果只保存“当前章节”，恢复时很容易丢失精确位置，尤其在滚动模式下更明显。

## EPUB CFI 是什么

`CFI` 可以粗略理解为一种“指向 EPUB 文档内部内容位置”的标准路径表示法。

它常用于：

- 书签
- 高亮
- 批注
- 精确恢复阅读位置
- 布局变化后的重定位

它的目标是：即使重新分页、字体变化，依然尽量能把用户定位回原文附近。

但对实现来说，CFI 的复杂度明显高于普通的 `href + fragment`。如果目标只是先把“目录跳转”和“上一章/下一章”做稳定，优先级通常应当是：

1. `opf`
2. `spine`
3. EPUB 3 导航文档 / `toc.ncx`
4. `href + fragment`
5. 最后再考虑 `CFI`

## 成熟阅读器通常会做的数据抽象

为了让目录跳转、阅读进度、书签和恢复位置都更稳定，阅读器一般会在解析阶段构建一层内部模型，而不是每次都直接从 HTML 临时解析。

一个常见的数据模型可能包括：

- `Book`
- `Resource`
- `SpineItem`
- `TocItem`
- `Locator`

例如：

```text
SpineItem
- id
- href
- mediaType
- linear

TocItem
- title
- href
- children

Locator
- spineIndex
- href
- fragment
- progress
```

有了这层中间模型后，很多功能都能共用同一套定位能力：

- 目录跳转
- 上一章/下一章
- 书签
- 阅读位置恢复
- 进度条拖动

## 章节跳转做稳还要处理哪些细节

### 1. 相对路径解析

EPUB 内部大量使用相对路径，例如：

```text
../Images/cover.jpg
../Styles/book.css
```

如果没有以当前文档路径为基准去解析，图片、样式、链接都可能失效。

### 2. fragment 定位

像 `chapter1.xhtml#p3` 这样的链接，`#p3` 必须在内容渲染完成后再定位。过早跳转通常会失败，尤其在 WebView 或自定义排版流程中更明显。

### 3. 嵌套目录

目录往往是多层结构，而不是一维列表。阅读器需要支持：

- 章节
- 小节
- 子小节

否则用户看到的目录层级会丢失。

### 4. 目录与正文不一致

有些 EPUB 会出现：

- 目录项存在，但目标文件不存在
- 目录项指向的锚点缺失
- `spine` 顺序与目录顺序不一致

成熟阅读器通常会做容错，而不是直接跳转失败。

### 5. 没有目录的 EPUB

这种情况下通常只能：

- 按 `spine` 实现上一项/下一项
- 尝试从标题标签中推断章节

但这种推断不一定可靠。

### 6. 分页模式与滚动模式差异

同样是“跳转到章节”，在不同阅读模式下底层实现并不相同：

- 滚动模式：更像滚到文档中的某个锚点
- 分页模式：更像打开某个资源，再重新分页并跳到对应页

所以很多阅读器的定位层不会直接存“页码”，因为页码对字体、窗口大小、主题和排版方式都很敏感。

## 如何判断一个阅读器的 EPUB 支持是否成熟

可以按能力层次理解：

### 第一层

能解压 EPUB，并按 `spine` 顺序显示内容。

### 第二层

能正确解析 EPUB 3 导航文档或 `toc.ncx`。

### 第三层

能稳定支持 `href + fragment` 的目录和文内跳转。

### 第四层

能准确保存并恢复阅读位置。

### 第五层

能支持 CFI、书签、高亮、批注。

### 第六层

能兼容质量较差的 EPUB，包括异常目录、错误锚点和脏数据容错。

很多让人感觉“章节跳转很完善”的阅读器，真正强的往往是第三层到第六层。

## 如果是为了自己实现阅读器，建议按这个顺序学习

1. 先理解 EPUB 文件结构
2. 再理解 `opf -> manifest / spine`
3. 再理解 EPUB 3 导航文档和 `toc.ncx`
4. 再理解 `href + fragment`
5. 最后再研究 `CFI`

不建议一开始就直接深入 CFI，因为它会把注意力过早拉到复杂定位细节，而不是先把主链路打通。

## 结论

阅读软件的“章节跳转很完善”，通常不是因为它能聪明地识别章节，而是因为它完整利用了 EPUB 自带的结构信息，并在内部建立了稳定的定位模型。

如果只看实现主线，可以把它归纳为一句话：

先解析 `opf` 建立资源和阅读顺序，再解析导航文档建立目录，最后通过 `href + fragment` 完成跳转，并用更细粒度的定位结构保存阅读位置。
