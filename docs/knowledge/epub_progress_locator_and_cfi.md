# EPUB 阅读位置、Locator 与 CFI

## 文档定位

这是一篇面向开发者的背景知识文档，用来解释阅读器为什么不能只保存“读到第几章”，以及 `Locator`、锚点、文本偏移、CFI 这些定位方式分别解决什么问题。

它的目标是：

- 帮助理解阅读位置为什么需要独立模型
- 帮助比较几种常见定位粒度
- 帮助理解 CFI 的价值和复杂度

## 为什么“读到第几章”通常不够

如果阅读器只保存：

- 当前章节

那么恢复时很容易出现：

- 只能回到章节开头
- 字体、窗口变化后位置偏差很大
- 同一正文文件内的小节无法准确恢复

因此成熟实现通常会保存更细粒度的位置。

## Locator 可以理解成什么

可以把 `Locator` 理解成“把用户当前位置表达成一组可恢复信息”的结构。

它不一定长成同一种样子，但通常会至少回答：

- 当前落在哪个正文单元
- 当前落在正文单元的什么位置
- 是否还能用更稳定的结构重新定位

## 常见定位粒度

### 1. `documentIndex + progress`

这是很多阅读器最容易先落地的一层。

例如：

```text
documentIndex = 12
progress = 0.42
```

优点：

- 简单
- 适合滚动模式
- 易于存储和恢复

缺点：

- 不是文本级精确定位
- 受正文高度、样式和布局影响

### 2. `href + anchor`

例如：

```text
Text/chapter8.xhtml#part3
```

优点：

- 语义直观
- 对目录跳转非常合适
- 适合有明确锚点的内容

缺点：

- 依赖锚点真实存在
- 不能覆盖所有位置

### 3. 文本偏移

例如：

- 某个文本节点中的字符偏移
- 某段内容中的相对字符位置

优点：

- 比纯滚动百分比更接近文本本身

缺点：

- 实现复杂
- 内容清洗或重排后容易漂移

### 4. EPUB CFI

它是一种更标准化的 EPUB 内部定位表示法。

优点：

- 适合精确定位
- 更适合书签、高亮、批注
- 布局变化后仍有机会重定位到原文附近

缺点：

- 学习和实现成本高
- 解析、生成、比较和重定位逻辑都更复杂

## CFI 解决的核心问题

CFI 试图解决的是：

- 位置不仅能存下来
- 还要尽量跨分页、跨字体变化、跨布局变化恢复

这也是为什么它常被用在：

- 高亮
- 批注
- 书签
- 精确恢复阅读位置

## CFI 不等于万能解

即使引入 CFI，也不代表：

- 所有 EPUB 都能完美恢复
- 所有内容变化都能无损重定位
- 目录跳转就不再需要 `href + fragment`

更现实的理解是：

- CFI 是更强的精确定位工具之一
- 但它通常建立在前面那几层基础能力已经可靠的前提上

## 为什么很多实现不会一开始就上 CFI

因为阅读器的主链路通常是：

1. 先把正文顺序建立起来
2. 再把 TOC 跳转建立起来
3. 再把基础阅读进度恢复做好
4. 最后才考虑文本级精确定位

如果主链路还不稳，过早引入 CFI 往往只会把复杂度提前。

## 一个实用的落地顺序

如果你是自己做阅读器，比较稳的顺序通常是：

1. `ReaderDocument`
2. `documentIndex + progress`
3. `href + anchor`
4. 书签和目录映射
5. CFI

这样做的好处是，前面的每一步都能单独形成可验证能力。

## 滚动模式与分页模式的影响

同样是“恢复到上次阅读位置”，不同模式下难点不同。

### 滚动模式

更容易先用：

- 文档级进度
- 锚点定位

### 分页模式

通常不能直接依赖“页码”，因为页码会随着：

- 字体大小
- 窗口尺寸
- 行距
- 主题和排版方式

发生变化。

这也是为什么很多实现会更倾向于保存“内容位置”，而不是保存“第几页”。
