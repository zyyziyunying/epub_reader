# EPUB TOC 模型与线性化

## 文档定位

这是一篇面向开发者的背景知识文档，用来解释 EPUB 的目录模型为什么天然是树，以及实现里为什么经常还要把它再线性化成一个稳定序列。

它的目标是：

- 帮助理解 EPUB 2 `toc.ncx` 和 EPUB 3 导航文档的共性
- 帮助理解树形 TOC 和线性顺序为什么要同时存在
- 帮助理解目录展示、跳转和持久化为什么常常需要一层中间模型

## TOC 在 EPUB 里到底是什么

TOC 可以理解为“给用户看的目录结构”，它的职责主要是帮助用户理解这本书的章节层级和入口，而不是单独定义正文阅读顺序。

它通常提供：

- 目录标题
- 指向的 `href`
- 父子层级关系

它通常不单独保证：

- 正文线性顺序一定正确
- 每个目录项都一定能成功跳转
- 一个目录项一定对应一个独立正文文件

## EPUB 2 与 EPUB 3 的目录来源

### EPUB 2

常见目录来源是 `toc.ncx`。

从高层看，它通常是一棵树：

- `navMap`
- `navPoint`
- 子 `navPoint`

每个节点一般会有：

- 标签文本
- 目标地址
- 子节点

### EPUB 3

常见目录来源是带 `nav` 属性的导航文档，通常是一个 XHTML 文档。

它常常以 HTML 列表来表达层级：

- `nav`
- `ol`
- `li`
- `a`

从实现角度看，它最后也会被还原成一棵树。

## 为什么 TOC 天然是树

一本书的目录往往天然是层级结构：

- 章
- 节
- 小节

如果实现里只保留一维列表，就会丢掉很多信息：

- 层级关系
- 父子结构
- 原始阅读上下文

这也是为什么“只要把目录读出来显示成一个数组”通常还不够。

## 为什么还要做线性化

虽然 TOC 天然是树，但很多功能又需要稳定的一维顺序：

- 把目录持久化到数据库
- 为每个目录项生成稳定 `order`
- 做“第一个命中的目录项”这类选择
- 在不直接渲染树的情况下做映射和回退

所以常见做法是：

- 保留原始树结构语义
- 同时生成一个稳定线性序列

## 常见的线性化方式

最常见的是前序遍历，也就是 pre-order DFS：

1. 先输出当前节点
2. 再按原始顺序递归输出它的子节点

这样做的好处是：

- 目录阅读顺序和用户感知通常比较一致
- 父节点天然出现在子节点前面
- 很容易稳定生成 `order`

如果原始 TOC 是：

```text
Chapter 1
  Section 1.1
  Section 1.2
Chapter 2
```

线性化后通常是：

```text
0 Chapter 1
1 Section 1.1
2 Section 1.2
3 Chapter 2
```

## 线性化后通常会保留哪些字段

常见字段包括：

- `title`
- `href`
- `order`
- `depth`
- `parentId`

这样一来：

- 树信息没有丢
- 一维顺序也稳定了

## 为什么 TOC 顺序和 spine 顺序可能不一样

这是 EPUB 很容易让人误解的地方。

TOC 更偏向“给用户看的结构入口”，而 `spine` 更偏向“正文阅读顺序单元”。

所以现实里可能出现：

- TOC 只列主要章节，没有列所有正文文件
- 多个 TOC 项指向同一个正文文件
- TOC 顺序和 `spine` 顺序不完全一致
- 某些目录项能展示，但跳转目标其实有问题

因此实现时不能简单假设：

- TOC 第 N 项 = 第 N 个正文文件

## 线性化对哪些能力最有帮助

它通常直接帮助下面这些事情：

- 稳定生成目录项 ID
- 做数据库持久化
- 判断“最早出现的目录项”
- 建立目录项到正文文档的映射
- 在缺少完整树渲染时做降级展示

## 常见实现误区

### 1. 只保留平面列表，不保留层级

这样目录抽屉也许能显示，但后续想做树形展示、折叠展开、面包屑定位时会很难补。

### 2. 运行时每次重新临时遍历

如果没有稳定 `order`，很多映射逻辑会变得脆弱。

### 3. 应用层自己重排 TOC

例如按标题字典序或按目标文件名排序。这样通常会破坏原书目录顺序。

### 4. 把 TOC 当成正文唯一来源

目录项是用户入口，不等于正文文档集合本身。

## 当没有 TOC 时怎么办

没有 TOC 的 EPUB 并不罕见。

这时常见退化策略包括：

- 使用 `spine` 作为基础顺序
- 尝试从正文标题推断导航项
- 明确接受“目录质量下降，但正文仍可读”

重点是：这类 fallback 应该被视为退化能力，而不是和原生 TOC 等价。
